# Анализ границ ответственности

**Система:** Умный дом «Lite» 🏠🌡️  
**Сценарий:** Регистрация показаний датчика и обработка алертов

---

## 1. Транзакционные границы

### 1.1. Где начинается и заканчивается транзакция?

**Начало транзакции:** Датчик отправляет POST-запрос с данными замера.

**Конец транзакции (успешный сценарий):**
- **Транзакция БД №1 (Персистентность):** Показание валидировано и сохранено в базу данных (TSDB).
- **Транзакция БД №2 (Алертинг):** Создана запись об инциденте (если порог превышен).
- **Побочные эффекты (асинхронные):** Отправка Push/Telegram уведомления пользователю и обновление данных на графиках в реальном времени.

**Важно:** Мы разделяем сохранение данных и уведомление. Если сервис уведомлений лежит, данные датчика всё равно должны быть сохранены.

### 1.2. Какие операции должны быть атомарными?

#### Транзакция №1: Прием и сохранение данных

**Атомарные операции:**
1. Проверка API-ключа датчика.
2. Валидация числового значения (диапазон, тип данных).
3. Запись замера в таблицу `sensor_readings`.
4. Обновление в кэше (Redis) значения "последнего полученного замера" для быстрого доступа.

**Обоснование:** Нельзя подтверждать датчику успех (201 Created), если данные не зафиксированы в постоянном хранилище.

#### Транзакция №2: Срабатывание алерта

**Атомарные операции:**
1. Сравнение замера с пороговым значением.
2. Создание записи в таблице `alerts` со статусом `NEW/TRIGGERED`.
3. Публикация события в очередь (Message Broker) для `Notification Worker`.

**Обоснование:** Если система зафиксировала превышение, запись об этом должна появиться в журнале инцидентов атомарно с постановкой задачи на отправку уведомления.

## 2. Таблица транзакционных границ

| Операция                              | Синхронная/Асинхронная | Откат при ошибке      | Retry-стратегия              | Идемпотентность            |
| ------------------------------------- | ---------------------- | --------------------- | ---------------------------- | -------------------------- |
| **Валидация JSON и ключа**            | Синхронная             | Нет (возврат 400/401) | N/A                          | Да                         |
| **Запись в TSDB (PostgreSQL)**        | Синхронная             | Да (ROLLBACK Tx1)     | 3 попытки (client-side)      | Да (по `sensor_id` + `ts`) |
| **Проверка порогов (Logic)**          | Синхронная             | Да (ROLLBACK Tx2)     | Нет                          | Да                         |
| **Запись инцидента в таблицу Alerts** | Синхронная             | Да (ROLLBACK Tx2)     | Нет                          | Да                         |
| **Публикация в Event Bus**            | Синхронная             | Да (ROLLBACK Tx2)     | Использование Outbox Pattern | Да                         |
| **Отправка Push-уведомления**         | Асинхронная            | Нет (данные уже в БД) | Exponential backoff (5 раз)  | Да (по `alert_id`)         |
| **Обновление веб-сокетов (графики)**  | Асинхронная            | Нет                   | 1 попытка (Best effort)      | Нет (не требуется)         |

## 3. Обработка отказов внешних сервисов

### 3.1. Notification Service (Telegram/Push) недоступен

**Проблема:** Критическая температура зафиксирована, но внешний API уведомлений не отвечает.

**Стратегия:**
1. **Таймаут:** 5 секунд.
2. **Действие:** Основная транзакция сохранения данных завершается успешно.
3. **Retry:** Сообщение остается в очереди (RabbitMQ/Kafka) или в таблице `notification_retry`.
4. **Уведомление:** При восстановлении сервиса уведомление уходит с пометкой "Задержка 5 минут".
5. **Fallback:** Если Telegram лежит более 30 минут, система может отправить резервное SMS (если настроено).

### 3.2. База данных временных рядов недоступна

**Проблема:** Хранилище показаний недоступно.

**Стратегия:**
1. **Таймаут:** 3 секунды.
2. **Действие:** Система возвращает датчику `503 Service Unavailable`.
3. **Retry:** Датчик обязан повторить запрос через 10, 30, 60 секунд.
4. **Важно:** Мы НЕ подтверждаем прием данных, чтобы датчик не удалил их из своей локальной памяти.

### 3.3. Кэш (Redis) с порогами недоступен

**Проблема:** Невозможно быстро проверить, превышен ли порог.

**Стратегия:**
1. **Fallback:** Система обращается напрямую в основную БД PostgreSQL в таблицу `sensor_settings`.
2. **Производительность:** Это медленнее, но гарантирует работу системы оповещений.

## 4. Идемпотентность

### 4.1. Проблема: Повторная отправка замера

**Сценарий:** Датчик отправил данные, БД их сохранила, но из-за сбоя сети датчик не получил ответ `201 Created` и шлет данные снова.

**Решение:**
1. Комбинированный уникальный ключ: `unique_reading_key = hash(sensor_id + timestamp)`.
2. В БД на колонку `timestamp` и `sensor_id` наложен `UNIQUE CONSTRAINT`.
3. При повторном запросе срабатывает `ON CONFLICT DO NOTHING`, и система возвращает `200 OK`.

## 5. Диаграмма потока данных (Transaction Flow)

```
[ Датчик ] 
    │ 
    │ 1. POST /readings (Value=35.0, Key=XYZ)
    ▼
[ API Gateway ] ─── (Auth Check) ───┐
    │                               │
    ▼                               ▼
[ Processing Service ] ────── [ TSDB (PostgreSQL) ] (Транзакция №1: Save)
    │                               │
    ├─ (Check Thresholds) ──────────┘
    │
    ▼
[ Event Bus (RMQ) ] ─── (Event: AlertTriggered) ───┐
    │                                               │
    ▼                                               ▼
[ Notification Worker ] ────────────── [ Notification Service (Telegram) ]
    │                                               │
    └────── (Retry if failed) ──────────────────────┘
```

## 6. Метрики и мониторинг

| Метрика               | Описание                                      | Критическое значение |
| --------------------- | --------------------------------------------- | -------------------- |
| `ingestion_latency`   | Время от получения данных до записи в БД      | > 500 мс             |
| `alert_delivery_time` | Время от фиксации превышения до отправки пуша | > 2 сек              |
| `db_error_rate`       | Процент неудачных попыток записи              | > 1%                 |
| `active_alerts_count` | Количество алертов в статусе PENDING          | > 100                |

## 7. Выводы

1. ✅ **Разделение ответственности:** Прием данных — критически важная синхронная операция. Отправка алертов — важная, но асинхронная операция.
2. ✅ **Надежность:** Использование очередей сообщений позволяет не терять уведомления при падении внешних API.
3. ✅ **Целостность:** Транзакционные границы гарантируют, что в системе не появится "фантомных" алертов без соответствующих данных в истории.
4. ✅ **Идемпотентность:** Реализована на уровне схемы БД, что предотвращает дублирование точек на графиках при нестабильной связи.
