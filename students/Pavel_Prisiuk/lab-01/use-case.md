# Use-case: Регистрация показаний датчика и обработка алертов

**Предметная область:** Система мониторинга «Умный дом lite» 🏠🌡️

**Первичный актор:** Датчик

**Вторичные акторы:**
- API Gateway (Входная точка)
- Processing Service (Сервис обработки данных)
- Database (База данных временных рядов / Time-series DB)
- Notification Service (Сервис уведомлений: Push, Telegram)
- Пользователь (Получатель алерта)

**Цель:** Принять данные от датчика, сохранить их для построения графиков и, если значение превышает норму, оперативно уведомить пользователя о критическом событии.

---

## Предусловия

- Датчик успешно зарегистрирован в системе и имеет уникальный `sensor_id` и `api_token`.
- У пользователя настроены пороговые значения (например, «Уведомить, если температура > 30°C»).
- Датчик имеет доступ к сети Интернет (Wi-Fi или Zigbee-шлюз).
- Сервис уведомлений и база данных доступны.

---

## Основной поток (Happy Path)

1. Датчик считывает текущий показатель (температура: 24.5°C).
2. Датчик формирует POST-запрос с JSON-пакетом: `{ "sensor_id": "temp-01", "value": 24.5, "timestamp": 1709635200 }`.
3. Система (API Gateway) проверяет авторизацию датчика по заголовку `X-API-KEY`.
4. Система проверяет валидность структуры данных:
   - `sensor_id` существует в реестре устройств.
   - `value` является числом в допустимом диапазоне (-50..+100 для температуры).
   - `timestamp` не старше 5 минут (защита от устаревших данных).
5. Система сохраняет показание в Базу Данных со статусом "PROCESSED".
6. Система извлекает из кэша (Redis) настройки алертов для данного датчика.
7. Система сравнивает полученное значение (24.5) с пороговым (30.0).
8. Условие алерта не выполняется (значение в пределах нормы).
9. Система публикует внутреннее событие `ReadingRegistered`.
10. Система возвращает датчику HTTP-ответ `201 Created`.
11. Пользователь видит обновленную точку на графике в мобильном приложении.

---

## Постусловия

- Показание успешно сохранено в БД с меткой времени.
- Графики в интерфейсе пользователя актуализированы.
- Транзакция по приему данных завершена.
- Уведомления не отправлялись (штатный режим).

---

## Альтернативные потоки

### 7a. Превышение порогового значения (Trigger Alert)

**Условие:** На шаге 7 система обнаруживает, что значение (35.0°C) выше порога (30.0°C).

**Поток:**
1. 7a1. Система создает запись в журнале инцидентов со статусом "TRIGGERED".
2. 7a2. Система инициирует асинхронный вызов Notification Service.
3. 7a3. Notification Service отправляет Push-уведомление пользователю: "Внимание! В гостиной жарко: 35.0°C".
4. 7a4. Система обновляет статус инцидента на "SENT".
5. 7a5. Возврат к шагу 10 основного потока.

### 5a. Дублирование данных (Idempotency Check)

**Условие:** Датчик отправил те же данные повторно из-за сетевого лага (не получил вовремя `201 Created`).

**Поток:**
1. 5a1. Система проверяет уникальный ключ запроса (или связку `sensor_id + timestamp`).
2. 5a2. Система обнаруживает, что данные для этой секунды от этого датчика уже записаны.
3. 5a3. Система НЕ создает новую запись в БД во избежание "шума" на графике.
4. 5a4. Система возвращает ответ `200 OK` (успешное подтверждение без повторной обработки).

---

## Исключительные ситуации

### 4a. Некорректный формат данных (Validation Error)

**Условие:** Датчик прислал невалидный JSON или значение вне физического диапазона (например, температура -500°C).

**Поток:**
1. 4a1. Система возвращает ошибку `400 Bad Request`.
2. 4a2. Система логирует попытку отправки некорректных данных (угроза безопасности или поломка датчика).
3. 4a3. Датчик получает ошибку и прекращает попытки до следующего цикла замера.

### 5b. База данных недоступна (Persistence Failure)

**Условие:** На шаге 5 БД не отвечает из-за перегрузки или аварии.

**Поток:**
1. 5b1. Система фиксирует `DatabaseConnectionException`.
2. 5b2. Система пытается повторить попытку записи (3 ретрая с экспоненциальной задержкой).
3. 5b3. Если запись не удалась, система возвращает ошибку `503 Service Unavailable`.
4. 5b4. Датчик (если поддерживает) кэширует значение в своей памяти для отправки позже.

### 7b. Критическое превышение при недоступном сервисе уведомлений

**Условие:** Температура выше нормы, но Notification Service вернул ошибку или таймаут.

**Поток:**
1. 7b1. Система сохраняет показание в БД (данные зафиксированы).
2. 7b2. Система создает запись об алерте со статусом "PENDING_RETRY".
3. 7b3. Система помещает задачу на отправку уведомления в очередь с гарантированной доставкой (например, RabbitMQ).
4. 7b4. Система возвращает датчику `201 Created` (так как данные успешно приняты).
5. 7b5. Как только Notification Service восстановится, пользователь получит алерт с пометкой о задержке.

---

## Частота использования

- **Высокая** - датчики отправляют данные каждые 30-60 секунд.
- **Пиковая нагрузка** - при резком изменении погоды или включении отопления (массовое срабатывание алертов).

## Критические требования

- **Идемпотентность:** Повторные пакеты данных не должны искажать статистику графиков.
- **Производительность:** Обработка одного замера (валидация + запись) должна занимать < 200 мс.
- **Надежность алертов:** Вероятность потери критического уведомления должна быть < 0.01%.
- **Целостность данных:** Нельзя подтверждать датчику прием данных (`201 Created`), если они не были физически записаны в БД.
- **Безопасность:** Защита от спуфинга (подмены ID датчика) через проверку токенов.

2. Как обрабатывать ситуацию, если датчик перестал выходить на связь?
3. Нужно ли хранить сырые JSON-логи запросов для отладки?
4. Должна ли система поддерживать групповые алерты (уведомлять всех членов семьи)?
